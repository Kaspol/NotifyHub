## Кто и как пользуется

В системе есть три основные роли:

- **Owner** — администратор организации.
- **Manager** — оператор уведомлений.
- **Viewer** — наблюдатель с доступом на чтение.

Пользователь сначала регистрируется или входит в систему, затем либо создаёт новую организацию, либо 
получает приглашение в существующую. После входа он попадает в рабочее пространство организации, 
а набор доступных действий зависит от роли: owner управляет участниками, ключами и webhooks, 
manager создаёт кампании и делает тестовые отправки, viewer только смотрит объекты и историю.

## Основной пользовательский сценарий

Нормальный сценарий использования должен выглядеть так:

1. Пользователь регистрируется и входит в систему.
2. Создаёт организацию или принимает приглашение.
3. Создаёт шаблон сообщения.
4. Создаёт список получателей.
5. Создаёт кампанию и связывает её с шаблоном и списком.
6. Делает `test-send`.
7. Если результат нормальный — планирует или запускает кампанию.
8. Смотрит delivery history, audit log и ошибки.

## Поведение системы по разделам

### 1. Аутентификация и сессии

С точки зрения пользователя auth работает так:

- На странице входа пользователь вводит email и пароль.
- Система проверяет учётные данные и выдаёт:
    - короткоживущий access token,
    - долгоживущий refresh token.
- Пока access token валиден, пользователь работает без повторного входа.
- Когда access token истекает, клиент автоматически обновляет его через refresh flow.
- При logout текущая сессия закрывается; в более зрелой версии пользователь может видеть список своих сессий 
и завершать их вручную.

Такой сценарий соответствует обычной модели short-lived access token + refresh token rotation, которую используют 
для API и SPA, чтобы уменьшать ущерб от утечки access token и не заставлять пользователя постоянно логиниться заново. 

### 2. Организация и роли

После входа пользователь всегда работает **в контексте организации**. Если у него несколько организаций, 
он выбирает активную, и весь интерфейс, поиск, фильтры, действия и API-запросы выполняются в рамках неё.

Поведение должно быть таким:

- В верхней части интерфейса видно, в какой организации ты находишься.
- Переключение организации меняет весь доступный набор данных.
- Пользователь не видит данные других организаций.
- Если роль не позволяет выполнить действие, кнопка либо скрыта, либо показывает понятную ошибку доступа.

Это не просто UX-деталь, а часть security-модели multi-tenant продукта: объектный доступ должен быть ограничен 
не только ролями, но и tenancy boundary, иначе возникают BOLA/IDOR-риски.

### 3. Шаблоны

Шаблон — это заготовка сообщения. Пользователь создаёт его один раз и потом использует в кампаниях.

Поведение:

- Пользователь задаёт название шаблона.
- Выбирает канал, например email или webhook payload template.
- Указывает subject и body.
- Добавляет переменные вроде `{{first_name}}`, `{{campaign_name}}`.
- Проверяет шаблон через preview с тестовыми данными.

Система должна вести себя предсказуемо: preview ничего не отправляет наружу, а только рендерит шаблон и показывает 
итог. Если обязательные переменные не заполнены, система сообщает, какие именно отсутствуют.

### 4. Списки получателей

Recipient list — это отдельный управляемый ресурс, а не просто вложенный массив в кампании.

Пользовательский сценарий:

- Создать список.
- Добавить получателей вручную или батчем.
- Хранить у каждого получателя email/phone/external_ref и attributes.
- Использовать один и тот же список в разных кампаниях.

Поведение системы:

- Список можно просматривать и редактировать отдельно.
- При удалении списка система предупреждает, где он используется.
- При тестовой отправке можно использовать либо весь список, либо 1–N тестовых адресов.

### 5. Кампании

Кампания — центральная сущность. Именно через неё пользователь "делает работу".

Поведение должно быть таким:

- Пользователь создаёт campaign в статусе `draft`.
- Привязывает template и recipient list.
- По желанию задаёт дату `scheduled_at`.
- Нажимает `test-send` для безопасной проверки.
- Если всё хорошо, запускает `launch` или `schedule`.

Жизненный цикл кампании:

- `draft`
- `scheduled`
- `running`
- `completed`
- `failed`
- `cancelled` (опционально)

Важно, чтобы программа не позволяла, например, запустить кампанию без шаблона или без аудитируемого контекста, 
а при повторном запуске не создавала неочевидное состояние. Пользователю должно быть ясно, можно ли редактировать 
кампанию в текущем статусе и какие действия доступны дальше.

### 6. Test send

`Test send` — отдельный сценарий, очень важный для UX.

Снаружи он должен работать так:

- Пользователь открывает кампанию.
- Нажимает "Send test".
- Выбирает 1–N тестовых получателей или использует специальный тестовый endpoint.
- Система сразу принимает запрос и создаёт задачу.
- Затем пользователь видит запись в `DeliveryAttempt` со статусом `queued/running/success/failed`.


### 7. Webhooks

Webhook endpoints — это внешние адреса, на которые NotifyHub отправляет события системы, например:

- `campaign.test_sent`
- `campaign.launched`
- `delivery.failed`
- `delivery.completed`

Со стороны пользователя поведение должно быть таким:

- Owner создаёт endpoint, указывая URL и набор event types.
- Система генерирует signing secret и показывает его один раз.
- Пользователь сохраняет этот secret у себя.
- Затем может сделать `test webhook`.
- В дальнейшем система отправляет события автоматически.

Очень важно, чтобы секрет показывался только при создании/ротации, а не в обычных GET-ответах, и чтобы у webhook 
была история delivery attempts. Для надёжных webhooks рекомендованы HMAC-подпись, idempotency, retry queue,
DLQ и manual replay, а обработчик на стороне клиента должен считать такие запросы недоверенными до успешной 
проверки подписи.

### 8. История доставок

Delivery history — это главный экран наблюдаемости для пользователя.

Пользователь должен видеть:

- какой объект отправлялся,
- куда,
- когда,
- какой был статус,
- сколько было попыток,
- какой ответ вернул endpoint,
- была ли доставка повторно отправлена.

Поведение системы:

- Каждая попытка доставки — отдельная запись.
- Повторы не "затирают" историю.
- Ошибка отображается понятным сообщением для пользователя и более подробным текстом в логах для инженера.
- В v3 можно дать manual replay для failed webhook deliveries.

Такой подход соответствует best practices для at-least-once delivery, retries, DLQ и event-level observability.

### 9. API keys

API keys нужны для машинного доступа, например CI, внешней интеграции или внутреннего сервиса.

Снаружи сценарий выглядит так:

- Owner создаёт API key.
- Указывает имя и scopes.
- Система показывает секрет только один раз.
- Дальше пользователь видит только prefix, дату создания, last used и статус.
- При компрометации ключ можно revoke или rotate.

Это должна быть понятная модель "создал → скопировал → больше полного значения не видишь". 
Такой UX важен, потому что API key — это credential, а не просто справочная запись.

### 10. Audit log

Audit log — это ответ на вопрос: **кто что сделал**.

Пользователь должен иметь возможность увидеть:

- кто создал кампанию,
- кто изменил шаблон,
- кто выпустил API key,
- кто сменил роль участнику,
- кто сделал test-send,
- кто удалил webhook endpoint.

Поведение:

- Audit log не редактируется.
- Записи упорядочены по времени.
- Есть фильтры по actor, action, resource type, date range.
- Sensitive события выделяются отдельно.

## Как пользоваться сервисом

Если описывать NotifyHub как инструкцию для пользователя, то порядок такой:

1. Войти в систему.
2. Создать или выбрать организацию.
3. Создать шаблон.
4. Создать список получателей.
5. Создать кампанию.
6. Сделать test-send.
7. Проверить delivery history.
8. Запустить кампанию.
9. При необходимости настроить webhooks и API keys.
10. Следить за audit log и статусами доставок.

То есть программа должна вести пользователя от настройки объектов к действию и от действия к наблюдаемому результату. 
Пользователь не должен "догадываться", что делать дальше: следующий шаг должен быть очевиден в интерфейсе и API-модели.

## Ошибки и реакции системы

Хорошая модель поведения включает не только happy path.

Программа должна вести себя так:

- При невалидных данных — вернуть понятную ошибку валидации.
- При отсутствии прав — вернуть отказ в доступе без утечки лишней информации.
- При истёкшем access token — попытаться обновить сессию через refresh.
- При недоступности webhook endpoint — записать failed delivery и поставить retry.
- При превышении лимита — вернуть `429`.
- При подозрительном URL webhook — отклонить регистрацию/валидацию.

Снаружи это должно ощущаться как "система предсказуемо объясняет, что произошло", а не как набор 500-ошибок. 
Для API-ориентированного продукта это особенно важно, потому что ошибки и поведение на границах напрямую влияют 
и на UX, и на безопасность.
